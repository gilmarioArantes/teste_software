%-------------------------------------------------------------------------Slide
\begin{frame}{O que \'e teste?}%18
    \begin{block}{Segundo a norma \cite{ieee610}}
    ``Teste  \'e o processo de executar um sistema ou componente sob condi\c{c}\~oes espec\'ificas, observando e registrando os resultados, avaliando alguns aspectos do sistema ou componente.''
    \end{block}
\end{frame}

%-------------------------------------------------------------------------Slide
\begin{frame}{Níveis de Maturidade em Teste de Software}%19
	\begin{block}{Segundo \cite{beizer1990}, são cinco:}
    	\begin{itemize}
    		\item Nível 0 - Não há diferença entre teste e depuração (debugging).
    		\item Nível 1 - O propósito do teste é mostrar que o software funciona.
    		\item Nível 2 - O propósito do teste é mostrar que o software não funciona.
    		\item Nível 3 - O propósito do teste não é provar nada, mas reduzir o risco de não funcionamento a um valor aceitável.
			\item Nível 4 - Teste não é uma ação, mas sim uma disciplina mental (institucionalizada na empresa) que resulta em software de baixo risco sem que seja empregado muito esforço de teste.
    	\end{itemize}
    \end{block}
\end{frame}

  \begin{itemize}

   \item Basicamente, os testes podem ser classificados em:
   \begin{itemize}

  \end{itemize}
\end{frame}

\section{Teste Funcional}
%===============================================================================

%-------------------------------------------------------------------------Slide
\begin{frame}{Critérios de Teste Funcional}%41
    \begin{enumerate}
        \item Particionamento de Equivalência (\textit{Equivalence Partition}).
        \item Análise do Valor Limite (\textit{Boundary Value Analysis}).
        \item Tabela de Decisão (\textit{Decision Table}).
        \item Grafo de Causa e Efeito (\textit{Cause-Effect Graph}).
        \item Teste de Todos os Pares (\textit{Pairwise Testing}).
        \item Teste de Transição de Estado (\textit{State-Transition Testing}).
        \item Teste de Análise de Domínio (\textit{Domain Analysis Testing}).
        \item Teste Funcional Sistemático (\textit{Systematic Functional Testing}).
        \item Teste de Caso de Uso (\textit{Use Case Testing}).
    \end{enumerate}
\end{frame}

%-------------------------------------------------------------------------Slide
\begin{frame}{Processo de Teste Caixa Preta}%42
 \begin{itemize}
  \item Os passos b\'asicos para se aplicar um crit\'erio de teste caixa preta s\~ao os seguintes:
  \begin{enumerate}
   \item A especifica\c{c}\~ao de requisitos \'e analisada.
   \item Entradas v\'alidas s\~ao escolhidas (com base na especifica\c{c}\~ao) para determinar se o produto em teste se comporta corretamente. 
   \item Entradas inv\'alidas tamb\'em s\~ao escolhidas para verificar se s\~ao detectadas e manipuladas adequadamente. 
   \item As sa\'idas esperadas para as entradas escolhidas s\~ao determinadas.
   \item Os casos de testes s\~ao constru\'idos.
   \item O conjunto de teste \'e executado.
   \item As sa\'idas obtidas s\~ao comparadas com as sa\'idas esperadas.
   \item \textcolor{red}{Um relat\'orio \'e gerado para avaliar o resultado dos testes.}
  \end{enumerate}
 \end{itemize}
\end{frame}

%-------------------------------------------------------------------------Slide
\begin{frame}{Teste Funcional e as Fases de Teste}%43
 \begin{itemize}
  \item Por ser independente da implementa\c{c}\~ao, crit\'erios da t\'ecnica caixa preta podem ser utilizados em todas as
fases de teste.
  \item A medida que se move do teste de unidade para o teste de sistema, entradas e sa\'idas mais complexas s\~ao exigidas, mas a abordagem permanece a mesma.
  \item Al\'em disso, principalmente no n\'ivel de sistema, o teste caixa-preta \'e de fundamental import\^ancia uma vez que a estrutura interna do sistema \'e muito complexa, inviabilizando o teste caixa-branca.
 \end{itemize}
\begin{figure}
 \includegraphics[width=8cm]{../imagens/fasesTesteFuncional.png}
\end{figure} 
\end{frame}

%-------------------------------------------------------------------------Slide
\begin{frame}{Vantagens do Teste Funcional}%44
 \begin{itemize}
  \item Pode ser utilizado em todas as fases de teste.
  \item Independente do paradigma de programa\c{c}\~ao utilizado.
  \item Teste funcional sistem\'atico direciona o testador a escolher subconjuntos de teste que s\~ao eficientes e efetivos em identificar defeitos. Melhor que o teste aleat\'orio \cite{Copeland2003}.
  \item Eficaz em detectar determinados tipos de defeitos:
  \begin{itemize}
    \item Funcionalidade ausente, por exemplo.
  \end{itemize}
 \end{itemize}
\end{frame}

%-------------------------------------------------------------------------Slide
\begin{frame}{Desvantagens do Teste Funcional}%45
 \begin{itemize}
  \item Dependente de uma boa especifica\c{c}\~ao de requisitos.
  \item N\~ao permite determinar que partes essenciais ou cr\'iticas da implementa\c{c}\~ao do software tenham sido executadas.
 \end{itemize}
\end{frame}

\section{Teste Estrutural}
%========================================================================SECTION

%-------------------------------------------------------------------------Slide
\begin{frame}{Teste Caixa Branca}%46
	\begin{itemize}
		\item Técnica Caixa Branca oposto da Técnica Caixa Preta.
		\item Baseia-se nos caminhos internos, estrutura e implementação do produto em teste.
		\item Requer conhecimento do código do produto em teste para ser aplicada.
	\end{itemize}
\end{frame}

%-------------------------------------------------------------------------Slide
\begin{frame}{Critérios de Teste Estrutural}%47
	\begin{itemize}
		\item Critérios Baseados em Fluxo de Controle:
		\begin{itemize}
			\item Todos-Nós, Todas-Arestas, Todos-Caminhos.
		\end{itemize}
		\item Critérios Baseados em Fluxo de Dados:
		\begin{itemize}
			\item Todas-Definições, Todos-P-Usos, Todos-C-Usos, Todos-Usos, Todos-Du-Caminhos, Todos-Pot-Usos,Todos-Pot-Du-Caminhos, Todos-Pot-Usos/Du.
		\end{itemize}
	\end{itemize}
\end{frame}

%-------------------------------------------------------------------------Slide
\begin{frame}{Processo do Teste Caixa Branca}%48
	\begin{itemize}
		\item Os passos básicos para se aplicar um critério de teste
caixa branca são os seguintes:
		\begin{enumerate}
			\item A implementação do produto em teste é analisada.
			\item Caminhos através da implementação são escolhidos.
			\item Valores de entradas são selecionados de modo que os caminhos selecionados sejam executados.
			\item As saídas esperadas para as entradas escolhidas são determinadas.
			\item Os casos de testes são construídos.
			\item Os casos de testes são executados.
			\item As saídas obtidas são comparadas com as saídas esperadas.
			\item Um relatório é gerado para avaliar o resultado dos testes.
		\end{enumerate}
	\end{itemize}
\end{frame}

%-------------------------------------------------------------------------Slide
\begin{frame}{Uso nas Fases de Teste}%49
	\begin{itemize}
		\item Critérios da técnica caixa branca também podem ser utilizados em todas as fases/níveis de teste.
		\item São mais comuns no teste de unidade e de integração.
		\item Teste de caminhos:
		\begin{itemize}
			\item Caminhos dentro de uma unidade.
			\item Caminhos entre unidades.
			\item Caminhos entre sub-sistemas.
			\item Caminhos entre o sistema todo.
		\end{itemize}
	\end{itemize}
\end{frame}

\begin{frame}{Vantagens do Teste Caixa Branca}%51
	\begin{itemize}
		\item Eficaz em determinar defeitos de lógica e de programação, especialmente no nível de unidade.
		\item É possível garantir que partes essenciais ou críticas do programa tenham sido executadas.
		\begin{itemize}
			\item Requisito mínimo de teste: garantir que o programa foi liberado tendo seus comandos executados ao menos uma vez por pelo menos um caso de teste.
		\end{itemize}
		\item É poss??vel saber o que ainda não foi testado.
	\end{itemize}
\end{frame}


%-------------------------------------------------------------------------Slide
\begin{frame}{Desvantagens do Teste Caixa Branca}%52
	\begin{itemize}
		\item O número de caminhos a serem executados pode ser infinito (semelhante ao teste exaustivo).
		\item O caso de teste selecionado pode não revelar o defeito sensível a dado. Por exemplo: 
		\begin{itemize}
			\item y = x * 2; // deveria ser y = x ** 2
			\item funciona corretamente para x = 0, y=0 e x = 2, y = 4.
		\end{itemize}				
		\item Assume fluxo de controle correto (ou próximo do correto). Casos de testes são baseados em caminhos existentes: caminhos inexistentes não podem ser descobertos.
		\item Determinação de caminhos não executáveis pode ser um problema.
		\item Dificuldade de automatização.
		\item \textcolor{red}{Habilidades de programação avançadas exigidas para compreender o código e decidir pela executabilidade ou não de um caminho.}
	\end{itemize}
\end{frame}

%-------------------------------------------------------------------------Slide
\begin{frame}{Níveis de Cobertura}%53
	\begin{itemize}
		\item Diferentes níveis de cobertura podem ser definidos em função dos elementos do GFC.
		\item Cobertura: porcentagem dos requisitos que foram testados versus o total de requisitos gerados.
		\item Oito diferentes níveis de cobertura são definidos por \cite{Copeland2003}.	
		\item Quanto maior o nível, maior o rigor do critério de teste, ou seja, mais caso de teste ele exige para ser satisfeito.
\begin{itemize}
	\item Nível 0 $\longleftarrow$ Nível 1 $\longleftarrow$ Nível 2 $\longleftarrow$ Nível 3 $\longleftarrow$ Nível 4 $\longleftarrow$ Nível 5 $\longleftarrow$ Nível 6 $\longleftarrow$ Nível 7
\end{itemize}
	\end{itemize}
\end{frame}

%-------------------------------------------------------------------------Slide
\begin{frame}{Níveis de Cobertura}%54
	\begin{itemize}
		\item Nível 0: qualquer valor de cobertura inferior a 100\% da cobertura de todos os comandos.
		\item Nível 1: 100\% de cobertura de comandos. (Todos-Nós)
		\item Nível 2: 100\% de cobertura de decisões. (Todos-Arcos)
		\item Nível 3: 100\% de cobertura de condições.
		\item Nível 4: 100\% de cobertura decisões/condições. 
		\item Nível 5: 100\% de cobertura de condições múltiplas.
		\item Nível 6: cobertura de loop.
		\item Nível 7: 100\% de cobertura de caminhos. (Todos-Caminhos)							
	\end{itemize}
\end{frame}
